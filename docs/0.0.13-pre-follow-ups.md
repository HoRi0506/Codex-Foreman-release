# v0.0.13-pre 추가 수정 작업 사항

## 1. 에이전트 이름 표기 불일치

- 관찰 증상: 사용자 화면에 나타나는 spawned agent line이 의도한 verifier 역할/이름과 분명히 맞지 않는다.
- 원하는 동작: visible output에서 역할과 실제 에이전트 이름이 바로 대응되어 보여야 한다.
- 구현 노트 / 수용 기준: task card, spawn log, transcript가 같은 역할 라벨을 사용해야 하며, verifier가 뜬 경우 사용자가 한눈에 verifier로 인지할 수 있어야 한다.

## 2. CCC lifecycle mutation 표시가 너무 verbose함

- 관찰 증상: CCC lifecycle 변경이 compact한 `ran` command output이 아니라 `Called` / MCP call 형태로 노출된다.
- 원하는 동작: operator-visible lifecycle mutation은 quiet CLI subcommand 중심으로 표시되어야 한다.
- 구현 노트 / 수용 기준: `ccc start`, `ccc orchestrate`, `ccc subagent-update`, `ccc memory` 같은 mutation 경로는 가능한 한 `ran` 형태로 기록되어야 하며, transcript에서 기본 노출이 MCP call 묶음으로 바뀌지 않아야 한다.

## 3. 중복 검증 흐름

- 관찰 증상: captain이 먼저 diff/status를 확인한 뒤 같은 검사를 verifier subagent에게 다시 보내서 중복 작업이 발생한다.
- 원하는 동작: 가능한 경우 검증은 직접 checking subagent로 라우팅되어 captain의 예비 확인과 중복되지 않아야 한다.
- 구현 노트 / 수용 기준: verification routing은 필요한 최소 단계만 거치고, 동일한 diff/status 확인이 captain과 verifier 사이에서 반복되지 않아야 한다. 동일 검증이 두 번 실행되지 않는 흐름이 기본이 되어야 한다.

## 4. `ccc subagent-update --quiet --json-file` payload 계약이 불명확함

- 관찰 증상: `ccc subagent-update --quiet --json-file <payload>`가 선호 quiet CLI 경로로 의도되었지만, JSON 파일이 실제로 존재하는데도 그럴듯한 payload가 `No such file or directory (os error 2)`로 실패했다.
- 원하는 동작: quiet CLI subcommand는 문서화된 payload shape를 그대로 수용하거나, 잘못된 경우 어떤 field/schema가 문제인지 명확하게 알려주는 validation error를 반환해야 한다. captain이 lifecycle update를 위해 MCP call로 되돌아갈 필요가 없어야 한다.
- 구현 노트 / 수용 기준: `ccc subagent-update --quiet --json-file`는 문서화되고 test-covered 된 schema를 가져야 하며, invalid payload는 actionable validation error를 내야 한다. valid fan-in/status payload는 CLI 경로로 처리되고 compact `ran` output으로 표시되어야 한다.

## 5. `ccc status --text --json-file`의 run-file payload 호환성/검증 문제

- 관찰 증상: `ccc status --text --json-file`가 run file을 대상으로 할 때 `Unexpected ccc_status argument: active_agent_id`로 실패한다.
- 원하는 동작: status CLI는 run-file payload의 필드 구성을 안정적으로 받아들이거나, 지원되지 않는 필드가 있으면 어떤 payload key가 호환되지 않는지 명확하게 설명해야 한다.
- 구현 노트 / 수용 기준: run-file 기반 status 조회는 payload shape 차이로 깨지지 않아야 하며, 호환되지 않는 field가 있더라도 validation error가 status CLI/run-file 계약 문제로 드러나야 한다.

## 6. `resolve_outcome=completed` 종료 전환이 verify task card를 다시 열 수 있음

- 관찰 증상: `ccc orchestrate`에서 `resolve_outcome=completed`를 지정해도 terminal 종료 대신 verify task card가 다시 선택되거나 열릴 수 있다.
- 원하는 동작: completed resolve는 terminal state로 바로 수렴해야 하며, 추가 verification 카드로 되돌아가지 않아야 한다.
- 구현 노트 / 수용 기준: 이 문제는 이미 문서화된 duplicate verification flow와 구분되어야 한다. `resolve_outcome=completed`는 completion/resolve state transition으로 처리되어 즉시 종료되고, verify card reopen/select가 발생하지 않아야 한다.

## 7. Planned LongWay row의 role/agent 표시가 실제 current task와 어긋날 수 있음

- 관찰 증상: planned LongWay row에 표시된 role/agent가 실제 current task와 다르게 보이거나 일관되지 않을 수 있다. 예를 들어 commit/push workflow에서 planned rows는 scribe/arbiter로 보였지만 current task는 companion_operator였다.
- 원하는 동작: planned-row display와 routing 표시는 실제 current task 및 실행 주체와 쉽게 대응되어야 한다.
- 구현 노트 / 수용 기준: planned rows는 display-only 정보와 routing state를 혼동시키지 않아야 하며, 사용자가 보기에 현재 task와 row role/agent가 불일치하지 않도록 명확한 매핑 또는 표기 규칙이 필요하다.

## 8. follow-up/release-card 문서 생성 후 추적 상태를 명확히 해야 함

- 관찰 증상: README-only commit 이후 생성된 follow-up 문서가 untracked 상태로 남을 수 있는데, 이때 workflow 상에서 이를 commit할지 pending으로 둘지 명시적으로 정리되지 않는다.
- 원하는 동작: 이 상황은 CCC runtime bug가 아니라 release-card workflow hygiene 문제로 다뤄져야 하며, follow-up 문서를 커밋할지 유지할지 결정하고 그 결과를 명확히 보고해야 한다.
- 구현 노트 / 수용 기준: release-card/follow-up workflow는 generated docs의 처리 방침을 명시적으로 선택해야 하고, 선택 결과가 tracked commit인지 pending artifact인지 사용자가 바로 알 수 있어야 한다.

## 9. host subagent thread limit 도달 시 fallback/reclaim 정책이 불명확함

- 관찰 증상: 이미 실행 중인 subagent가 있는 상태에서 추가 specialist를 spawn하려고 하면 `agent thread limit reached`가 발생할 수 있다. 이때 captain이 조용히 직접 구현으로 넘어가면 `$cap`의 specialist-owned 작업 원칙과 lifecycle 기록이 흐려진다.
- 원하는 동작: host subagent 슬롯/스레드 한도에 걸린 경우 CCC가 이를 명확한 lifecycle/fan-in 상태로 기록하고, 대기, 기존 subagent 회수/종료, 재시도, 명시적 fallback 중 하나를 선택해야 한다.
- 구현 노트 / 수용 기준: spawn 실패 원인이 thread limit이면 `host_subagent_thread_limit` 같은 구체적인 fallback reason으로 기록할 수 있어야 한다. captain이 직접 구현 fallback을 수행하는 경우에도 해당 fallback reason과 active subagent 정리 결과가 run/task card에 남아야 한다.

## 10. 완료된 host subagent thread 정리 상태가 CCC 상태와 어긋날 수 있음

- 관찰 증상: CCC 내부 상태에서는 active subagent가 0으로 보이지만, host Codex 쪽에는 완료된 subagent thread가 닫히지 않고 남아 추가 spawn 슬롯을 점유할 수 있다.
- 원하는 동작: subagent가 completed/failed/stalled/merged/reclaimed 같은 terminal 상태가 되면 CCC가 host thread cleanup 필요성을 명확히 드러내고, 가능한 경우 close/release를 유도하거나 기록해야 한다.
- 구현 노트 / 수용 기준: CCC status와 lifecycle 기록은 `active_handle_cleanup`뿐 아니라 host에 남아 있는 terminal thread cleanup 필요성도 구분해서 보여야 한다. thread limit fallback이 발생한 경우 기존 completed subagent를 닫았는지, 닫지 못했는지, 어떤 수동 조치가 필요한지 operator가 바로 알 수 있어야 한다.
